# План реализации SuperAI

План пересобран по зависимостям и проверяемым вертикальным срезам. Номер файла задаёт порядок работы, а не номер архитектурного модуля.

1. [00-architecture-decisions.md](./00-architecture-decisions.md) — границы системы, термины, контракты и правила принятия решений.
2. [01-client-first.md](./01-client-first.md) — диагностический клиент, mock-трассы и первый сквозной сценарий.
3. [02-runtime-and-storage.md](./02-runtime-and-storage.md) — API, событийный runtime, трассировка, объектное хранилище и архив.
4. [03-hive-memory.md](./03-hive-memory.md) — контракт задачи, Улей, горячая/тёплая память, вытеснение и восстановление.
5. [04-cosmos-and-execution.md](./04-cosmos-and-execution.md) — Космос, импорт, Атлас, фабрики, семьи, навыки, планировщик и критики.
6. [05-learning-and-evolution.md](./05-learning-and-evolution.md) — компостирование, компиляция опыта, геномы и эволюция.
7. [06-roadmap-and-validation.md](./06-roadmap-and-validation.md) — этапы, gates, MVP, метрики и системные тесты.
8. [07-legacy-reuse.md](./07-legacy-reuse.md) — что переносить из `master` и `alg2`, а что оставить исследовательским прототипом.

## Как пользоваться планом

- Работа идёт по одному вертикальному срезу до его gate, а не сразу по всем модулям.
- Клиент начинается первым и до появления backend работает на тех же версионированных DTO и mock-трассах.
- Следующий этап не начинается, пока предыдущий не наблюдаем через клиент и не покрыт соответствующими тестами.
- Объёмы данных, задержки и проценты задаются после базовых измерений на конкретном компьютере. Они не заменяют проверку корректности.
- Эксперименты из старых веток переносятся через адаптеры и characterization-тесты; целиком ветки не сливаются.

## Порядок первого результата

```text
контракты
→ клиент на mock-трассе
→ минимальный API
→ один Улей в памяти
→ детерминированный ответ
→ живая трасса в клиенте
→ сохранение и восстановление
→ малый Космос
→ планирование и проверка
→ переработка опыта
```
