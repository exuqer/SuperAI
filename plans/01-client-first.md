# 01. Клиент первым и первый вертикальный срез

## Цель

С первого дня иметь рабочий инструмент, через который можно запускать сценарий, видеть трассу и сравнивать ожидаемое поведение с фактическим. До появления backend клиент воспроизводит версионированные mock-трассы; затем источник данных переключается на live API без замены экранов.

Клиент не содержит доменную логику Космоса или Улья. Он показывает состояние, причины решений и безопасно инициирует команды.

## Стек

Использовать проверенный стек из `master`:

- Vue 3;
- TypeScript;
- Vite;
- Vue Router;
- Pinia;
- SCSS;
- Vitest и Vue Test Utils;
- Playwright;
- Cytoscape только для экранов, где граф действительно нужен.

Каркас `web/` из `master` переносится выборочно по плану из [07-legacy-reuse.md](./07-legacy-reuse.md). Статический клиент `alg2/web` не брать за основу.

## Этап 1. Shell и режим mock

Реализовать:

- общий `AppShell`;
- маршруты `Run`, `Traces`, `Hive`, `Storage`, `Cosmos`, `System`;
- переключатель `mock/live` только для development;
- каталог сохранённых fixtures;
- общие состояния loading, empty, error, offline и stale;
- отображение версии API и схемы fixture.

Не создавать двадцать пустых вкладок по будущим модулям. Новая страница появляется вместе с диагностическим сценарием и реальными данными.

## Этап 2. Экран запуска

Минимальная форма:

- сообщение пользователя;
- `conversation_id` и необязательный `project_id`;
- бюджет времени и событий;
- кнопки запуска и отмены;
- результат, ошибки и список использованных источников;
- ссылка на `trace_id`.

На mock-этапе submit выбирает fixture по сценарию. После подключения API тот же use case вызывает `POST /api/v1/tasks`.

## Этап 3. Универсальный инспектор трассы

Инспектор должен показывать:

- хронологию span/event;
- дерево причинности `causation_id`;
- компонент, входы, выходы, длительность и статус;
- изменения бюджета;
- ссылки на артефакты;
- нормализованную ошибку и решение о retry;
- сравнение двух запусков.

Фильтры:

- component;
- kind;
- status;
- task/hive;
- интервал времени;
- только критический путь.

Первая реализация — таблица, timeline и JSON inspector. Граф причинности добавляется после появления достаточно сложной трассы.

## Этап 4. Инспекторы состояния

### Hive

- контракт задачи;
- активные цели и ограничения;
- горячая память и заполнение бюджета;
- магазины;
- вытесненные элементы с причиной;
- доступные снимки и операция восстановления.

### Storage

- метаданные артефакта;
- владельцы и ссылки;
- checksum и статус целостности;
- область доступа;
- история версий;
- безопасный preview с ограничением размера.

### Cosmos

- понятия и утверждения;
- источник и точный фрагмент;
- противоречия;
- оценки и объяснение их изменения;
- фильтры сектора и времени;
- локальный граф вокруг выбранного узла.

### System

- health/readiness;
- версия backend и схем;
- активные задачи;
- очередь незавершённой работы;
- dead letters;
- базовые метрики и последние ошибки.

## Клиентская модель данных

Разделить слои:

```text
transport DTO
→ mapper/validator
→ UI model
→ page/widget
```

UI не импортирует внутренние Python-структуры и не предполагает расположение данных на диске. Входной DTO валидируется на границе; неизвестные поля игнорируются, неизвестная major-версия показывает ошибку совместимости.

## API первого среза

Минимальные endpoints:

```text
GET  /api/v1/health
GET  /api/v1/meta
POST /api/v1/tasks
POST /api/v1/tasks/{task_id}/cancel
GET  /api/v1/tasks/{task_id}
GET  /api/v1/traces/{trace_id}
GET  /api/v1/hives/{hive_id}
GET  /api/v1/artifacts/{artifact_id}/metadata
```

Долгие операции возвращают `202` и статус задачи. Для первого локального среза клиент может опрашивать статус с backoff; SSE/WebSocket добавляется, только если polling мешает измеряемому сценарию.

## Fixtures

Каждый fixture содержит запросы, ответы API, временную шкалу и ожидаемое состояние экрана. Обязательные сценарии:

- успешный ответ;
- ошибка валидации контракта;
- превышение бюджета;
- восстановимый сбой и retry;
- отмена пользователем;
- несовместимая версия схемы;
- backend недоступен.

Fixtures хранятся рядом с contract-тестами и проверяются backend-схемой в CI.

## Тесты

- unit-тесты mapper/validator;
- component-тесты shell, маршрутов и инспекторов;
- contract-тесты всех fixtures;
- e2e: запустить mock-задачу, открыть трассу, перейти к Улью и артефакту;
- e2e live: health, один детерминированный запрос, отображение той же трассы;
- доступность клавиатурной навигации и узкий viewport;
- ошибка backend не уничтожает уже открытую сохранённую трассу.

## Gate 1

- Клиент запускается независимо от backend.
- Все эталонные fixtures проходят один и тот же валидатор.
- По `trace_id` можно восстановить цепочку от запроса до результата.
- Mock и live используют одинаковые DTO.
- Один реальный минимальный backend-обработчик виден в интерфейсе без ручного чтения логов.
