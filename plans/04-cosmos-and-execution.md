# 04. Космос и исполняемые возможности

## Результат этапа

Небольшой контролируемый корпус импортируется с происхождением, запрос извлекает ограниченную область Космоса, а планировщик внутри текущего Улья собирает и проверяет маршрут из зарегистрированных возможностей.

## 1. Космос утверждений

### Каноническая модель

```text
Concept
├── concept_id, canonical label, type
└── aliases and language forms

Claim
├── subject_id, predicate, object/value
├── provenance refs
├── valid time and scope
├── access scope
├── verification status
└── independent scores
```

Противоречащие claims хранятся рядом; один не перезаписывает другой. Удаление источника пересчитывает только зависящие от него свидетельства.

Секторы — сохранённые представления/фильтры общего графа. Узел не дублируется при принадлежности нескольким секторам.

### Индексы MVP

- индекс идентификаторов и типов;
- полнотекстовый поиск label/alias;
- adjacency по типизированным отношениям;
- простой векторный индекс как необязательный кандидатный поиск;
- фильтры источника, времени и доступа.

Финальный retrieval score раскладывается на признаки и трассируется. Векторное сходство не считается доказательством отношения.

## 2. Архив, карантин и импорт

Пайплайн:

```text
Source received
→ immutable archive artifact
→ quarantine record
→ parse factories
→ candidate concepts/claims
→ validation and deduplication
→ critic reports
→ atomic integration or rejection
```

Первые источники: `.txt` и Markdown из небольшого fixture-корпуса.

Первые фабрики:

- document loader;
- structural splitter;
- sentence splitter;
- dictionary/entity extractor;
- simple claim extractor;
- source binder;
- concept resolver.

Каждая фабрика является чистой или идемпотентной относительно входа, возвращает типизированный результат и не записывает напрямую в чужой агрегат.

### Обновление и удаление

- повторный импорт того же content hash ничего не дублирует;
- новая версия источника создаёт diff;
- старые claims получают корректный статус, а не исчезают без следа;
- удаление по политике доступа проходит через все производные индексы;
- непроверенный импорт не виден обычному retrieval по умолчанию.

## 3. Retrieval

Вход: TaskContract, контекст Улья, разрешённые области доступа и бюджет.

Выход:

- компактный набор claims;
- ссылки на точные источники;
- scores и причины включения;
- отмеченные противоречия и пробелы;
- затраченный бюджет.

Порядок MVP: lexical candidates → filters → graph expansion с ограничением → rerank → diversity/dedup. Radius без limit и фильтров доступа запрещён.

## 4. Атлас возможностей

Атлас регистрирует не предметные ульи, а capability manifests:

```text
capability_id, version, kind
input/output schemas
supported operations/modalities
preconditions and access needs
quality/reliability statistics
latency/cost/load estimates
artifact/code reference
dependencies and health
```

Виды capability: factory, family, skill, critic, codec, bridge, retriever.

Атлас отвечает на вопрос «что может преобразовать этот тип в требуемый результат в данном бюджете?». Он не исполняет маршрут и не решает, какой Улей продолжить.

## 5. Фабрики и семьи

`Factory` выполняет конкретное типизированное преобразование. `Family` координирует одну ограниченную операцию или подзадачу, используя разрешённые фабрики и магазины.

В первой версии семья — обычный обработчик с локальным состоянием и бюджетом. Отдельные объекты-муравьи добавляются только в эксперименте, где популяционный поиск сравнивается с более простой базовой стратегией.

Минимальные семьи:

- retrieval;
- planning;
- answer assembly;
- verification.

## 6. Процедуры и навыки

Операция описывает input/output schema, предусловия, эффект, стоимость и обработку ошибки. Навык — версионированный граф операций с тестами и статистикой.

Первый набор операций формируется из MVP-сценария, например:

```text
LOAD_SOURCE → FIND_CLAIMS → FILTER → COMPARE
→ VERIFY_PROVENANCE → SUMMARIZE → FORMAT
```

Не вводить универсальный `EXECUTE` без типизированного эффекта: такой узел скрывает границы безопасности и делает план непроверяемым.

## 7. Планировщик

Поток:

```text
TaskContract + Hive state
→ goal graph
→ Atlas candidates
→ bounded route candidates
→ static validation
→ execute best candidate
→ critics
→ local repair or fail
```

Первая стратегия — детерминированный ограниченный поиск по типам и стоимости. Beam search добавляется после появления сценария, где greedy/bounded shortest path объективно проигрывает.

Ограничения:

- предел глубины и числа кандидатов;
- запрет capability без подходящей версии схемы;
- явные side effects;
- циклы только с условием остановки;
- план сохраняется до выполнения;
- любое перепланирование создаёт новую ревизию.

## 8. Критики

Для MVP:

- `ContractCritic` — соответствует ли результат цели и формату;
- `ProvenanceCritic` — есть ли допустимые источники у значимых утверждений;
- `ConsistencyCritic` — нет ли внутренних противоречий;
- `BudgetCritic` — не скрыто ли превышение;
- `ContextCritic` — нет ли примеси другой темы или области доступа.

Критик возвращает report с target, severity, evidence, repair hint и verdict. Генератор и критик логически разделены; отдельный процесс или модель нужны только по профилю риска.

## 9. Текстовый кодек

Первая версия строит ответ из проверенного смыслового плана детерминированными шаблонами. После декодирования выполняется обратная проверка:

- все обязательные тезисы присутствуют;
- новых claims не появилось;
- ссылки на источники сохранились;
- язык и формат соответствуют контракту.

Подключение внешней или локальной LLM допускается как capability за явным интерфейсом, с записанной версией модели, prompt/template ref и обязательной проверкой результата. Архитектура не должна предполагать, что собственный генератор естественного языка уже решён.

## 10. Мосты

Мост добавляется, если различаются схемы или представления, например `table rows → statistical series` или `scene graph → text plan`. Использование двух секторов одного графа само по себе моста не требует.

Мост сообщает mapping, потери, неопределённость, версию и provenance.

## Тестовый корпус

Начать с 30–100 коротких документов и gold-набора:

- ожидаемые concepts/claims;
- неоднозначные слова;
- два противоречащих источника;
- устаревшее утверждение;
- удаление одного источника;
- вопрос без ответа в корпусе;
- вопрос, требующий двух секторов;
- попытка доступа к чужому private claim.

Сначала измерять precision/recall на gold-наборе и полноту provenance. Цель «100 000 понятий» не является gate: большой граф может скрыть ошибки модели.

## Gate 4

- Импорт идемпотентен и обратим по источнику.
- Значимые утверждения ответа имеют разрешённое provenance либо явно помечены как гипотезы.
- Противоречия не схлопываются в одно число или одну строку.
- Атлас выбирает возможности, а не предметные ульи.
- Новый составной маршрут собирается, проверяется и полностью виден в клиенте.
- Retrieval читает ограниченную область и соблюдает budget/access filters.
