# 02. Runtime, наблюдаемость и хранилище

## Результат этапа

После перезапуска система восстанавливает подтверждённое состояние и незавершённую работу, не выполняет успешно завершённую команду второй раз и позволяет открыть полную трассу в клиенте.

## 1. API и composition root

На базе FastAPI создать тонкий транспортный слой:

- валидация и аутентификационный контекст;
- преобразование HTTP DTO в command/query;
- единая модель ошибок;
- composition root зависимостей;
- OpenAPI как источник клиентских типов;
- отсутствие доменной логики в route handlers.

Для локальной разработки backend может раздавать собранный клиент, но dev-server клиента остаётся отдельным.

## 2. Event runtime

### Ответственность

- реестр обработчиков команд и событий;
- очередь запланированной работы;
- приоритеты и `scheduled_at`;
- бюджеты времени, памяти и количества шагов;
- отмена;
- тайм-аут;
- ограниченные retry с backoff и jitter;
- dead-letter запись;
- восстановление незавершённой работы.

### Не входит

- бизнес-решение задачи;
- глобальный цикл опроса всех компонентов;
- гарантия exactly-once на внешние эффекты.

Реальная гарантия MVP: at-least-once доставка плюс идемпотентный обработчик и transactional outbox для связи изменения состояния с публикацией события.

### Минимальная модель

```text
WorkItem
├── command_id, task_id, trace_id
├── handler, payload/schema version
├── status, attempt, scheduled_at
├── idempotency_key
├── budget snapshot
└── last_error
```

Состояния: `queued → running → succeeded | failed | cancelled | dead_letter`.

События завершённых задач не «проигрываются заново» как команды после перезапуска. Восстанавливаются материальные состояния и только незавершённые work items.

### Порядок реализации

1. Синхронный in-process dispatcher.
2. Очередь SQLite и один worker.
3. Отмена, deadline и event/step budget.
4. Идемпотентность и outbox.
5. Retry-классификация и dead letters.
6. Несколько workers только после concurrency-тестов.

## 3. Наблюдаемость

Для каждого запроса сохранять:

- структурированную трассу;
- ключевые доменные события;
- метрики длительности и размера;
- изменения бюджета;
- решения выбора с оценками кандидатов;
- ошибки без секретов и больших payload.

`TraceSpan` содержит `trace_id`, `span_id`, `parent_span_id`, компонент, операцию, ссылки на вход/выход, status, start/end, budget before/after и error ref.

Лог не является источником доменного состояния. Трасса имеет ограниченную политику хранения; точные входы лежат в архиве с собственной областью доступа.

## 4. Типизированное объектное хранилище

### Слои

```text
ArtifactRepository     — тип, версия, владелец, связи, жизненный цикл
BlobStore              — байты по content hash
Metadata/Transaction DB — SQLite в MVP
Archive                — неизменяемые исходники и версии
SnapshotStore          — согласованные снимки агрегатов
```

### ArtifactRef

```text
artifact_id
content_hash
media_type
schema_name/schema_version
size
tenant_id
created_at
encryption/access metadata
```

`artifact_id` обозначает логический артефакт, `content_hash` — конкретное содержимое. Это позволяет версионировать объект и при этом дедуплицировать одинаковые байты.

### Правила записи

- запись идёт во временный файл;
- вычисляются размер и hash;
- файл атомарно публикуется;
- метаданные и outbox фиксируются транзакцией;
- повтор с тем же idempotency key возвращает прежний результат;
- чтение проверяет hash по политике целостности.

### Сборка мусора

Не полагаться только на счётчики ссылок: они ломаются при циклах и частичных сбоях. Использовать mark-and-sweep от версионируемого набора корней: активные агрегаты, снимки, архивные политики и pinned-артефакты. Удаление сначала помечает объект и даёт grace period.

## 5. Сериализация

Первая версия:

- JSON для контрактов и небольших типизированных объектов;
- SQLite для метаданных и очереди;
- файлы для blob;
- checksum и schema version обязательны.

Компактную бинарную сериализацию вводить после профилирования и с миграционным тестом. Собственный бинарный формат не является gate MVP.

## 6. Снимки и восстановление

- Снимок связан с последней применённой последовательностью событий/команд.
- Создание снимка атомарно относительно агрегата.
- При старте проверяется совместимость схемы.
- Повреждённый последний снимок не уничтожает предыдущий.
- Восстановление выполняется в отдельном тестовом каталоге и сравнивается с ожидаемым hash состояния.

## Тесты

### Runtime

- порядок очереди и starvation guard;
- отмена до и во время обработки;
- deadline и каждый вид бюджета;
- временная/постоянная ошибка;
- crash между изменением состояния и публикацией outbox;
- повтор command с одинаковым idempotency key;
- перезапуск при `queued` и `running` work item.

### Storage

- одинаковые bytes используют один blob;
- разные логические версии имеют отдельные artifact records;
- повреждение обнаруживается;
- авария в каждой точке записи не публикует полуартефакт;
- mark-and-sweep сохраняет достижимый цикл и удаляет недостижимый;
- tenant boundary запрещает чужое чтение;
- снимок восстанавливается побитово или семантически согласно контракту.

### Observability

- все spans связаны в одну трассу;
- секретные поля редактируются;
- клиент открывает live-трассу и dead letter;
- sampling не удаляет обязательный audit trail.

## Gate 2

- Детерминированный сквозной запрос переживает перезапуск.
- Успешный внешний эффект не повторяется при повторной доставке.
- Повреждённый blob и несовместимая схема обнаруживаются до использования.
- Отмена и превышение бюджета видны клиенту с точной причиной.
- Восстановление проверено fault-injection тестами, а не только штатным завершением.
