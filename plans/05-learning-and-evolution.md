# 05. Переработка опыта, навыки и эволюция

## Условие начала

Этот этап начинается только когда предыдущие gates пройдены и накоплены воспроизводимые трассы с результатами критиков. Без стабильных контрактов, baseline и отката автоматическое «обучение» лишь закрепит ошибки.

## 1. Перегнивание контекста

Завершённый или вытесненный контекст разбирается на кандидаты:

- подтверждённые claims;
- гипотезы;
- решения и их область применимости;
- успешные процедуры;
- отрицательный опыт;
- пользовательские предпочтения;
- точные архивно значимые фрагменты;
- вычислимые дубликаты и шум.

Каждый кандидат сохраняет ссылки на исходную трассу и область доступа. Классификатор не имеет права превратить гипотезу в подтверждённое утверждение.

## 2. Компостирование

Компост — компактный производный артефакт, а не удаление исходника. Он содержит:

```text
type and normalized content
source trace/artifact refs
applicability scope
access scope
confidence and verification status
reconstruction pointer
compiler/version
```

Интеграция в Космос — отдельная команда после проверок. Приватный компост остаётся приватным. Обобщение между пользователями требует отдельной обезличивающей процедуры и явной политики.

## 3. Компилятор опыта

### Пайплайн

1. Отобрать успешные трассы одного класса задач.
2. Нормализовать конкретные значения в типизированные параметры.
3. Найти повторяющиеся подграфы операций.
4. Сформировать кандидата навыка с предусловиями.
5. Запустить на train-наборе для отладки.
6. Сравнить с baseline на отложенном наборе.
7. Провести canary или shadow-прогон.
8. Опубликовать новую версию или отклонить.

Не компилировать маршрут только по числу повторений. Нужны минимальное разнообразие входов и отсутствие data leakage между обучающим и проверочным набором.

### Критерий пользы

```text
utility = quality_delta
          - α * latency_delta
          - β * resource_delta
          - γ * maintenance_cost
          - risk_penalty
```

Снижение числа событий не гарантирует реального ускорения. Измерять wall time, CPU, peak RAM, I/O и качество на фиксированном окружении.

### Жизненный цикл навыка

`candidate → validated → shadow → active → deprecated → retired`.

Для каждой версии сохраняются manifest, procedure graph, test set refs, метрики, compiler version, provenance и rollback target.

## 4. Консолидация

Фоновая работа:

- дедупликация производных артефактов;
- переиндексация;
- пересчёт явно определённых scores;
- проверка устаревших навыков;
- создание снимков;
- поиск кандидатов компоста и навыков;
- применение retention policy.

Консолидация запускается как обычная бюджетированная задача runtime. После неё выполняются инвариантные и regression-тесты. При ухудшении новые версии не продвигаются.

## 5. Геномы

Геном нужен только для исполняемого компонента, который действительно имеет повторяемую структуру и варианты. Минимальная модель:

```text
genome_id and version
component type
module/operation refs
parameters and interface schema
parent refs and derivation recipe
operator versions
resulting content hash
evaluation refs
```

Состояние сессии, горячая память и содержимое конкретного Улья не входят в геном.

Начать с одного родителя и патча. Два родителя, crossover и архетипы добавлять после доказанной экономии хранения или качества поиска.

## 6. Генеалогический DAG

Различать:

- provenance DAG — фактическое происхождение;
- storage DAG — оптимизированное хранение общих блоков.

Оптимизация storage DAG не должна переписывать историю provenance. Материализация проверяется по content hash; периодические полные снимки ограничивают длину цепочки патчей.

## 7. Эволюционный движок

Первый объект эволюции — небольшой детерминированный маршрутизатор или набор параметров, а не вся система.

Цикл:

```text
baseline
→ isolated candidates
→ mutation
→ fixed benchmark
→ Pareto comparison
→ shadow evaluation
→ manual/automatic promotion policy
```

Качество, задержка, память, размер и риск не сводить без необходимости в один непрозрачный score. Хранить Pareto frontier и объяснение выбора.

Эволюция не выполняет произвольный код и не получает production-секреты. Все side effects отключены или sandboxed, бюджеты жёсткие, набор тестов версионирован.

## 8. Мультимодальность

Начинается после стабильного текстового MVP. Первая модальность — та, для которой есть измеримый сквозной сценарий; предпочтительно простая структурная графика или обработка уже готовых изображений.

Общий интерфейс кодека:

```text
encode(raw artifact) → typed semantic artifacts
decode(semantic artifacts) → raw artifact
evaluate(original, reconstructed, task contract) → metric report
```

Один универсальный loss для текста, изображения и аудио не вводится.

## Тесты

- компост восстанавливается до заявленного уровня;
- provenance и access scope не теряются при обобщении;
- гипотеза не становится verified claim;
- кандидат навыка не видит holdout во время компиляции;
- новая версия не ухудшает baseline сверх допуска;
- rollback возвращает прежние ответы на regression-наборе;
- геном материализуется с ожидаемым hash;
- повреждённый patch обнаруживается;
- фоновая консолидация не нарушает latency активной задачи;
- удаление пользовательских данных затрагивает производные private artifacts.

## Gate 5

- На серии разнообразных задач навык даёт измеримое улучшение без статистически/практически значимого падения качества.
- Результат воспроизводим на чистом окружении.
- Любая опубликованная версия имеет provenance, benchmark и rollback.
- Геномы и эволюция остаются опциональными: отсутствие выигрыша не блокирует полезный MVP.
