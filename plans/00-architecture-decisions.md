# 00. Архитектурные решения и границы

## Цель первой версии

Первый MVP не должен доказывать, что система заменяет большую языковую модель. Он должен воспроизводимо проверить более узкие гипотезы:

- активный контекст ограничен и не загрязняется старой темой;
- нужная область знаний извлекается локально и с происхождением;
- новый маршрут собирается из известных операций;
- результат и причина каждого решения наблюдаемы;
- повторяющийся маршрут после отдельной проверки можно превратить в версионируемый навык;
- восстановление после перезапуска и откат не теряют подтверждённые данные.

## Единый словарь

| Термин | Значение | Не является |
| --- | --- | --- |
| Космос | глобальный граф понятий и утверждений с происхождением | большим текстом или единственным векторным индексом |
| Сектор | логическое представление части Космоса | отдельной копией знаний |
| Улей | рабочее пространство диалога, проекта или задачи | предметной базой знаний |
| Атлас | каталог данных и исполняемых возможностей | менеджером ульев или решателем задачи |
| Фабрика | типизированное преобразование | автономным планировщиком |
| Магазин | временное типизированное хранилище Улья | долговременным источником истины |
| Семья | ограниченный исполнитель операции или подзадачи | постоянно работающей группой процессов |
| Навык | проверенный версионируемый граф процедур | записью конкретного диалога |
| Мост | явное преобразование схемы, представления или модальности | связью между любыми двумя темами |
| Геном | воспроизводимое описание версии исполняемого компонента | снимком состояния Улья |

Следствия:

- `HiveManager` решает, какой рабочий Улей продолжить, создать или восстановить.
- Атлас выбирает кандидатов фабрик, семей, навыков, критиков, кодеков и представлений Космоса.
- Один Улей может использовать сколько угодно секторов в пределах бюджета.
- «PythonHive», «BiologyHive» и подобные постоянные предметные ульи в коде не создаются.

## Архитектурный стиль MVP

Начать с модульного монолита на Python и отдельного web-клиента. Внутренние пакеты имеют явные интерфейсы, но выполняются в одном процессе, пока измерения не покажут необходимость изоляции.

Разделять взаимодействия на:

- **command** — просьба изменить состояние, имеет один ответственный обработчик;
- **query** — чтение без скрытого изменения состояния;
- **domain event** — неизменяемая запись уже произошедшего значимого факта.

Не нужно отправлять событие для каждого локального вызова функции. События нужны для наблюдаемости, асинхронной работы, восстановления и слабой связанности на границах компонентов.

## Канонические идентификаторы

Минимальный набор:

```text
tenant_id
user_id
project_id
conversation_id
hive_id
task_id
trace_id
span_id
event_id
artifact_id
schema_version
```

Идентификатор не должен одновременно обозначать задачу, диалог и Улей. Все записи содержат область доступа; глобальные данные маркируются явно.

## Канонические контракты

До реализации backend зафиксировать JSON Schema или OpenAPI для:

- `TaskContract`;
- `CommandEnvelope`;
- `DomainEvent`;
- `TraceSpan`;
- `ArtifactRef`;
- `ModuleSnapshot`;
- `CriticReport`;
- `ErrorEnvelope`;
- `AnswerEnvelope`.

Обязательные поля конверта:

```text
id, schema_version, occurred_at
tenant_id, task_id, trace_id
kind, producer
payload или artifact_ref
causation_id, correlation_id
```

Тяжёлые данные передаются через `artifact_ref`. Маленький payload допускается прямо в конверте; запрет на любой inline payload только усложнит отладку.

## Версионирование и совместимость

- Схема имеет явную версию.
- Неизвестная major-версия отклоняется с понятной ошибкой.
- Совместимое расширение не ломает старого читателя.
- Миграции тестируются на сохранённых fixtures.
- TypeScript-типы клиента генерируются из OpenAPI/JSON Schema или проверяются contract-тестом. Дублировать контракты вручную можно только на самом первом mock-этапе.

## Данные и достоверность

Космос хранит утверждения, а не неоспоримые факты. Для утверждения обязательны:

- субъект, отношение и объект;
- ссылка на источник и при возможности точный фрагмент;
- время получения и период действия;
- область применимости и область доступа;
- статус проверки;
- независимые оценки достоверности, релевантности, полезности, свежести и противоречивости.

Повторение повышает частоту и, возможно, полезность, но не достоверность. Embedding и полнотекстовый индекс являются перестраиваемыми индексами, а не каноническим состоянием.

## Надёжность и безопасность

- Обработчики команд идемпотентны по ключу операции.
- Повторы ограничены; необработанные команды попадают в dead-letter запись с причиной.
- Нельзя логировать секреты и полные приватные payload по умолчанию.
- Удаление данных пользователя распространяется на архив, индексы, компост и обученные приватные артефакты согласно политике хранения.
- Автоматическое обучение не публикует пользовательские данные в общую область.
- Любое обученное изменение имеет происхождение, оценку до/после и путь отката.

## Стандарт документации компонента

Для реально реализуемого компонента достаточно:

```text
docs/components/<component>/
├── README.md
├── contracts.md
├── decisions/
└── operations.md
```

Документация должна отвечать на вопросы: ответственность и границы, входы/выходы, состояние, ошибки и восстановление, наблюдаемость, бюджет, тесты и текущий статус. Не создавать пустые шаблоны для всех будущих модулей заранее.

## Gate 0

- Термины не противоречат основному описанию.
- Есть версионированные схемы минимального сквозного сценария.
- Есть пять эталонных трасс: success, validation error, timeout, retry, cancellation.
- Для каждого артефакта понятны владелец, область доступа и срок жизни.
- Зафиксированы ADR о модульном монолите, хранилище MVP и способе генерации клиентских типов.
