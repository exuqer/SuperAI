# Пониматель

`/api/understand` и страница `/understand` нужны не для ответа пользователю, а для диагностики первого шага разбора текста.

Задача слоя:

`текст -> raw tokens -> рабочая форма -> search token -> concept uri -> статус совпадения`

Он работает read-only: ничего не пишет в checkpoint, не трогает чат и не запускает муравьиный поиск.

## Что именно происходит

Для каждого слова модуль делает несколько последовательных шагов:

1. Берет `raw_token` из исходного текста обычной токенизацией по словам.
2. Определяет язык. Если язык `ru`, подключается `pymorphy3`.
3. Для русского слова `pymorphy3` дает морфологический разбор и грамматические теги.
4. Затем модуль строит набор рабочих форм для поиска. Для уменьшительных и похожих форм он может отрезать короткий суффикс вроде `ик`, `чик`, `очк`, `еньк`, `ышк`, `ушк`.
5. Из этих форм выбирается первая подходящая рабочая форма. Именно она попадает в поле `lemma`.
6. По `lemma` строится `search_token`.
7. Дальше идет поиск по checkpoint: алиасы, метки концептов, описания, а потом легкие эвристики.
8. Если совпадение не найдено, токен остается `candidate`, но все равно получает синтетический `concept_uri` вида `/c/ru/кот`.

В seed-словаре уже есть точечные алиасы для стабильных слов и имен, например `кушать -> /c/ru/есть` и `яна -> /c/ru/яна`.

## Почему `котик` превращается в `кот`

Это не чистый стемминг и не простое удаление окончания. Логика такая:

1. `pymorphy3` разбирает `котик` как отдельное слово и дает морфологическую информацию.
2. Затем модуль смотрит на несколько рабочих вариантов.
3. Для форм на `-ик` он пробует более короткую основу.
4. В результате рабочая форма становится `кот`.

То есть `lemma` в этом модуле означает не «строгую словарную лемму», а «форму, удобную для поиска в памяти».

Пример:

```text
котик -> кот
котики -> кот
котиками -> кот
```

## Что возвращается

Каждый токен содержит:

- `raw_token` - исходное слово;
- `lemma` - рабочая нормализованная форма;
- `search_token` - ключ для поиска;
- `concept_uri` - URI найденного или синтетического концепта;
- `match_status` - как именно токен был распознан;
- `is_stop_word` - является ли токен стоп-словом;
- `morphology` - `POS`, `case`, `number`, `gender`, `tense`, `person`, если они доступны.

## Статусы совпадения

- `found_as_alias` - совпало с алиасом из checkpoint;
- `found_as_lemma` - совпало по рабочей форме;
- `found_as_raw` - совпало по сырой форме;
- `partial_root_match` - найден близкий общий корень;
- `edit_distance_match` - найдено через небольшую опечатку;
- `candidate` - прямого совпадения нет, токен идет как кандидат;
- `ignored_stop_word` - токен выкинут как шум.

## Практический пример

Для запроса:

```text
котики едят
```

слой обычно возвращает:

- `котики -> кот`
- `едят -> есть`

Если в checkpoint есть подходящие алиасы, статус будет `found_as_lemma`, `found_as_alias` или `found_as_raw`. Если алиаса нет, статус останется `candidate`, но `search_token` все равно будет использоваться дальше как рабочий ключ.

## Что этот слой не делает

- не пишет в checkpoint;
- не меняет чатовые сессии;
- не использует эмбеддинги;
- не строит ответ пользователю;
- не подменяет полный анализ смысла, а только готовит токены для последующих слоев.
