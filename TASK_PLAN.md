ЗАДАЧА: РЕАЛИЗОВАТЬ МНОГОУРОВНЕВЫЙ ИЕРАРХИЧЕСКИЙ УЛЕЙ

Нужно расширить существующую систему SuperAI и добавить построение предложений и новых словоформ сверху вниз:

смысл сцены
→ роли предложения
→ лексемы
→ словоформы
→ морфемы
→ буквы.

Не переписывай текущую архитектуру с нуля. Сначала изучи существующий backend, модели базы данных, API, алгоритм обучения, физику улья, JSON-экспорт и frontend.

Существующие типы scene, concept_candidate, lexeme, word_form, character, а также пространства global_field, hive_space, scene_space и word_structure_space должны продолжить работать.

Главный результат этапа:

1. Улей должен уметь раскрывать внутреннюю структуру активированной лексемы или словоформы.
2. При отсутствии готовой словоформы система должна попытаться собрать её из морфологических шаблонов и букв.
3. Новая форма сначала должна существовать только как гипотеза внутри улья.
4. Глобальная долговременная память не должна загрязняться каждым сгенерированным вариантом.
5. Предложение должно строиться сначала как смысловой и ролевой каркас, а не сразу как строка символов.
6. После генерации система должна перечитать свой ответ снизу вверх и проверить его соответствие исходному смыслу.

==================================================
1. ОСНОВНЫЕ АРХИТЕКТУРНЫЕ ПРИНЦИПЫ
==================================================

1.1. Не помещать все буквы всех активных слов в основной улей.

Основной улей сохраняет ограничение max_cells, например 24 смысловые ячейки.

Буквы, морфемы и варианты словоформ должны находиться во вложенных пространствах со своими лимитами и не занимать основные ячейки улья напрямую.

1.2. Использовать ленивое раскрытие уровней.

Обычный путь:

concept/scene
→ lexeme
→ существующая word_form
→ готовая строка.

Раскрывать морфемы и буквы только когда:

- готовая словоформа не найдена;
- несколько форм имеют близкий ранг;
- нужно проверить неизвестную форму;
- пользователь или аналитический интерфейс вручную открыл внутренний уровень;
- выполняется генерация нового слова.

1.3. Разделить долговременные и временные объекты.

В глобальном поле могут храниться:

- подтверждённые лексемы;
- подтверждённые словоформы;
- подтверждённые морфемы;
- устойчивые морфологические шаблоны;
- устойчивые рамки предложений.

Только внутри улья должны существовать:

- hypothesis_word_form;
- sentence_slot;
- generated_sentence;
- временные морфемные комбинации;
- отклонённые варианты слов.

1.4. Не превращать систему в обычный семантический граф.

Структурные отношения допустимы:

- слово состоит из букв;
- форма принадлежит лексеме;
- морфема занимает определённую позицию;
- слот предложения реализован определённой словоформой.

Но не создавать глобальные статические рёбра для каждого смыслового соседства.

Смысловые совпадения должны по-прежнему определяться пространством, локальной активацией, ролями, гравитацией и резонансом.

==================================================
2. НОВАЯ ИЕРАРХИЯ
==================================================

Реализовать следующую структуру:

Уровень 0. Сцена или замысел ответа

Пример:

scene:
  predicate: есть
  subject: котик
  object: рыба

Уровень 1. Слоты предложения

subject:
  lexeme: котик
  number: plural
  case: nominative

predicate:
  lexeme: есть
  tense: present
  person: third
  number: plural

object:
  lexeme: рыба
  number: singular
  case: accusative

Уровень 2. Выбранные словоформы

котик + plural/nominative → котики
есть + present/third/plural → едят
рыба + singular/accusative → рыбу

Уровень 3. Морфемная структура

котик + и
ед + ят
рыб + у

Уровень 4. Буквенная поверхность

к о т и к и
е д я т
р ы б у

Результирующее предложение:

Котики едят рыбу.

==================================================
3. ИЗМЕНЕНИЯ ТИПОВ ОБЪЕКТОВ
==================================================

Добавить подтверждённые глобальные типы облаков:

- morpheme_candidate;
- morpheme;
- morph_operator;
- morph_pattern;
- sentence_frame.

Не создавать глобальный тип hypothesis_word_form. Это временный тип узла улья, а не постоянное облако глобального поля.

Назначение типов:

morpheme_candidate:
повторяющийся фрагмент, смысл и функция которого ещё недостаточно подтверждены.

morpheme:
подтверждённый устойчивый фрагмент, например корень, суффикс или окончание.

morph_operator:
преобразование, которое необходимо выполнить:

- PLURAL;
- SINGULAR;
- NOMINATIVE;
- ACCUSATIVE;
- PAST;
- PRESENT;
- THIRD_PERSON;
- DIMINUTIVE;
- NEGATION;
- DERIVATION.

Не присваивать оператору семантическое имя DIMINUTIVE только по наличию суффикса «ик». Такой смысл должен быть подтверждён контекстами или обучающей разметкой.

morph_pattern:
устойчивое правило преобразования поверхности.

Примеры:

основа + PLURAL → «ы»;
основа после ж/ч/ш/щ/к/г/х + PLURAL → «и»;
основа «рыб» + ACCUSATIVE_SINGULAR → «у»;
лексема на «ик» + PLURAL → «ики».

sentence_frame:
порядок и совместимость ролей предложения.

Пример:

subject + predicate + object;
predicate + object + location;
location + predicate + subject.

==================================================
4. НОВЫЕ ТИПЫ ПРОСТРАНСТВ
==================================================

Добавить:

morphology_space:
внутреннее пространство лексемы.

Оно содержит:

- известные словоформы;
- морфемы;
- морфологические операторы;
- применимые шаблоны;
- статистику подтверждений.

sentence_frame_space:
пространство синтаксического каркаса предложения.

hive_subspace:
временное вложенное пространство конкретной ячейки улья.

Существующий word_structure_space сохранить.

Разница:

morphology_space отвечает на вопрос:
«Какие формы может принимать эта лексема и почему?»

word_structure_space отвечает:
«Из каких символов состоит конкретная словоформа?»

Пример:

lexeme «рыба»
└── morphology_space
    ├── word_form «рыба»
    ├── word_form «рыбы»
    ├── word_form «рыбу»
    ├── stem «рыб»
    ├── ending «а»
    ├── ending «ы»
    └── ending «у»

word_form «рыбу»
└── word_structure_space
    ├── р
    ├── ы
    ├── б
    └── у

6. ОБУЧЕНИЕ МОРФОЛОГИИ
==================================================

Расширить текущий pipeline обучения.

Для каждого токена:

1. Определить или создать lexeme.
2. Определить или создать word_form.
3. Сохранить доступные грамматические признаки.
4. Создать или обновить morphology_space лексемы.
5. Поместить известную word_form в morphology_space.
6. Сохранить буквенную последовательность в word_structure_space.
7. Сравнить новую форму с другими формами той же лексемы.
8. Найти стабильную часть и изменяемую часть.
9. Создать или усилить morpheme_candidate.
10. Создать или усилить morph_pattern.

Для первого этапа использовать детерминированное сравнение строк:

- общий префикс;
- общий суффикс;
- различающийся хвост;
- нулевое окончание;
- позицию различия;
- последние буквы основы;
- длину основы.

Примеры:

рыба / рыбу
→ стабильная часть «рыб»
→ замена «а» на «у».

котик / котики
→ стабильная часть «котик»
→ добавление «и».

мяч / мячи
→ стабильная часть «мяч»
→ добавление «и».

Не пытаться на первом этапе полностью решить русскую морфологию, чередования и исключения.

Сначала реализовать ограниченный, объяснимый и расширяемый механизм.

==================================================
7. ОБНАРУЖЕНИЕ МОРФЕМ
==================================================

Морфема не должна становиться подтверждённой после одного совпадения.

Первоначально создавать morpheme_candidate.

Учитывать:

- количество разных лексем, где встретился фрагмент;
- одинаковую позицию фрагмента;
- одинаковые грамматические условия;
- похожий результат преобразования;
- успешность дальнейшего применения;
- число отклонённых форм.

Пример условного перехода:

morpheme_candidate
→ morpheme

при выполнении условий:

- evidence_count >= конфигурируемого порога;
- встречается более чем в одной лексемной семье;
- confidence >= порога;
- число успешных применений выше числа ошибок.

Пороги вынести в конфигурацию.

Для небольшого демонстрационного датасета разрешить низкий тестовый порог, но production-настройки должны быть строже.

==================================================
8. ФОРМИРОВАНИЕ ПРЕДЛОЖЕНИЯ
==================================================

Не генерировать ответ сразу текстом.

Сначала создать sentence_plan внутри улья.

Пример:

{
  "frame": "subject_predicate_object",
  "slots": [
    {
      "role": "subject",
      "concept_cloud_id": "...",
      "lexeme_cloud_id": "...",
      "requested_features": {
        "number": "plural",
        "case": "nominative"
      }
    },
    {
      "role": "predicate",
      "lexeme_cloud_id": "...",
      "requested_features": {
        "tense": "present",
        "person": 3,
        "number": "plural"
      }
    },
    {
      "role": "object",
      "concept_cloud_id": "...",
      "lexeme_cloud_id": "...",
      "requested_features": {
        "number": "singular",
        "case": "accusative"
      }
    }
  ]
}

Для каждого слота выполнить:

1. Найти лексему.
2. Определить требуемые грамматические признаки.
3. Проверить наличие готовой word_form.
4. При точном совпадении использовать её.
5. При отсутствии формы раскрыть morphology_space.
6. Активировать совместимые morph_pattern.
7. Создать несколько гипотез.
8. Собрать буквы каждой гипотезы.
9. Оценить варианты.
10. Выбрать устойчивую форму.

После выбора словоформ собрать предложение в порядке sentence_frame.

Добавить:

- регистр первой буквы;
- пробелы;
- базовую пунктуацию.

==================================================
9. ГЕНЕРАЦИЯ НЕИЗВЕСТНОЙ СЛОВОФОРМЫ
==================================================

Алгоритм:

resolve_word_form(lexeme, requested_features)

Шаг 1:
найти точную известную форму.

Шаг 2:
найти формы этой лексемы с наиболее близкими признаками.

Шаг 3:
активировать шаблоны, совместимые с:

- частью речи;
- концом основы;
- требуемым оператором;
- ролью предложения;
- контекстом сцены.

Шаг 4:
создать top-K вариантов.

Шаг 5:
каждый вариант представить как hypothesis_word_form только внутри улья.

Шаг 6:
создать буквенное пространство варианта.

Шаг 7:
провести прямую и обратную проверку.

Пример:

запрос:
lexeme = «мяч»
number = plural

варианты:

мячи;
мячы;
мяча;
мячики.

При обычном PLURAL должна победить форма «мячи».

Форма «мячики» не является простым множественным числом слова «мяч».

Она требует двух операций:

мяч
→ DERIVATION/DIMINUTIVE
→ мячик
→ PLURAL
→ мячики.

Поэтому не генерировать «мячики», если в смысловом плане нет операции уменьшительности или перехода к лексеме «мячик».

Если значение суффикса «ик» ещё не изучено, система может хранить структурный оператор DERIVATION_SUFFIX_ИК без утверждения, что он означает уменьшительность.

==================================================
10. ОЦЕНКА ВАРИАНТОВ
==================================================

Реализовать объяснимый итоговый score.

Пример компонентов:

score_total =
    semantic_compatibility
  + grammatical_compatibility
  + morph_pattern_support
  + orthographic_support
  + contextual_support
  + reverse_validation
  - contradiction_penalty
  - unsupported_operation_penalty.

Весовые коэффициенты вынести в конфигурацию.

Не называть итоговый score вероятностью.

Для каждого кандидата сохранять объяснение:

- какой оператор применён;
- какой шаблон использован;
- на каких примерах он обучен;
- какие признаки совпали;
- какие признаки не подтверждены;
- почему вариант проиграл.

Пример:

«мячи»

- PLURAL совпадает;
- шаблон для основы на «ч» поддержан;
- лишних derivation-операторов нет;
- обратный разбор возвращает лексему «мяч».

«мячики»

- PLURAL совпадает;
- добавлен неподтверждённый derivation-оператор;
- смысл уменьшительности отсутствует;
- применить штраф.

==================================================
11. ОБРАТНАЯ ПРОВЕРКА
==================================================

После сборки словоформы выполнить:

буквы
→ предполагаемая word_form
→ lexeme
→ грамматические признаки
→ роль предложения.

После сборки предложения выполнить:

текст
→ компоненты
→ роли
→ сцена
→ сравнение с исходным sentence_plan.

Проверять:

- сохранилась ли выбранная лексема;
- сохранились ли число, падеж, время и лицо;
- сохранилась ли роль;
- не появился ли новый неподтверждённый смысл;
- не потерян ли обязательный слот;
- нет ли конфликта согласования.

Если reverse_validation_score ниже порога:

- не выводить вариант как стабильный;
- повысить температуру соответствующего подпространства;
- выполнить дополнительный шаг вибрации;
- активировать следующий кандидат.

==================================================
12. МНОГОУРОВНЕВАЯ ВИБРАЦИЯ
==================================================

Не вибрировать все уровни одинаково.

Добавить конфигурируемые базовые частоты по типам:

- scene: низкая частота, высокая инерция;
- concept: низкая или средняя частота;
- lexeme: средняя частота;
- word_form: выше средней;
- morpheme: высокая;
- morph_pattern: средняя или высокая;
- character: самая высокая частота и быстрое затухание.

Начальные значения можно задать в конфигурации, например:

scene: 0.6
concept: 0.8
lexeme: 1.0
word_form: 1.2
morpheme: 1.4
character: 1.7

Не считать эти значения окончательными.

Реализовать передачу энергии в двух направлениях.

Сверху вниз:

смысл и роль
→ активируют совместимую лексему
→ лексема активирует формы
→ форма активирует морфемы
→ морфемы активируют буквы.

Снизу вверх:

совместимая буквенная сборка
→ усиливает морфемную комбинацию
→ усиливает word_form
→ подтверждает лексему и роль.

Несовместимый нижний уровень должен ослаблять верхнюю гипотезу.

Каждое вложенное пространство имеет отдельный энергетический бюджет.

Не раскрывать все доступные варианты. Использовать top-K:

- до 5 словоформ;
- до 8 морфологических комбинаций;
- до 3 буквенных вариантов на комбинацию.

Значения сделать конфигурируемыми.

==================================================
13. СТАБИЛИЗАЦИЯ
==================================================

Не считать процесс стабильным только после одного шага.

Для основного улья и вложенных пространств проверять сходимость.

Пример условий:

- порядок top-кандидатов не меняется несколько шагов;
- изменение общей энергии ниже порога;
- средняя скорость узлов ниже порога;
- выбранная словоформа не меняется;
- reverse_validation_score выше порога;
- несколько шагов не было вытеснений.

Пример:

STABLE, если в течение 3 последовательных шагов:

delta_energy < 0.02
delta_ranking < 0.03
mean_velocity < 0.01
reverse_validation_score >= 0.75

Пороговые значения вынести в конфигурацию.

Основной улей может быть стабилен, даже если нераскрытое внутреннее пространство не запускалось.

Если внутреннее пространство участвует в генерации ответа, его статус должен учитываться в общем статусе.

19. КРИТЕРИИ ГОТОВНОСТИ
==================================================

Этап считается выполненным, когда:

1. Текущий чат и обычный поиск продолжают работать.

2. Любую активную словоформу можно раскрыть до букв.

9. «Мяч → мячик → мячики» реализуется как DERIVATION + PLURAL.

10. Предложение сначала существует как sentence_plan.

11. После генерации выполняется обратный разбор.

12. Интерфейс позволяет переходить между уровнями.

13. Вибрация разных уровней использует разные параметры.

14. Стабилизация требует нескольких шагов сходимости.

Не пытаться сразу реализовать:

- полную русскую морфологию;
- все исключения;
- все части речи;
- сложные чередования;
- совершенный синтаксический анализ;
- длинную свободную генерацию текста;
- обучение значения суффиксов исключительно по буквам.

- существительные;
- единственное и множественное число;
- несколько базовых падежных форм;
- простые формы глаголов;
- subject/predicate/object/location;
- простое словообразование через отдельный оператор;
- сборку коротких предложений.

Внешний морфологический анализатор не должен становиться источником ответа системы.

При необходимости его можно использовать только:

- для тестовой разметки;
- для сравнения результатов;
- в режиме отладки.

В runtime итоговая форма должна определяться собственной памятью, шаблонами и резонансом улья.
