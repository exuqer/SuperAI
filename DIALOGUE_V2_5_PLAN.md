# SuperAI V2.5: план естественного диалога и безопасной языковой интерпретации

## 1. Цель V2.5

V2.5 превращает текущий вопросно-ответный контур SuperAI V2 в устойчивую систему
русскоязычного диалога.

Система должна:

1. Разбирать составную реплику на коммуникативные акты, предложения и клаузы.
2. Извлекать несколько связанных событий из одной реплики.
3. Различать утверждение, вопрос, просьбу, команду, предположение, условие, цитату,
   подтверждение, отрицание, уточнение и исправление.
4. Хранить конкурирующие морфологические, синтаксические, ролевые и референциальные
   гипотезы.
5. Принимать решение по независимым доказательствам.
6. Разрешать местоимения, эллипсис и ссылки по локальному состоянию диалога.
7. Запрашивать уточнение при критической неоднозначности.
8. Поддерживать смену темы, возврат к теме и исправление предыдущего хода.
9. Разделять историю разговора, обязательства говорящих, языковую память и
   подтверждённое знание о мире.
10. Формировать краткие, полные, уточняющие, неопределённые и конфликтные ответы.
11. Сохранять происхождение каждого решения от исходного текста до готового ответа.
12. Поддерживать отзыв, замену и повторную интерпретацию знания.

Главный принцип:

> Реплика создаёт локальные гипотезы интерпретации. Устойчивая интерпретация обновляет
> состояние диалога. Подтверждённое знание проходит отдельный строгий допуск в
> долговременную память.

## 3. Архитектурные решения

### 3.1. Клауза становится основной единицей интерпретации

Структура текста:

```text
диалог
└── реплика
    ├── предложение
    │   ├── клауза
    │   └── клауза
    └── предложение
        └── клауза
```

Каждая клауза получает:

- собственный предикат;
- участников;
- коммуникативный режим;
- фактуальность;
- полярность;
- область отрицания;
- модальность;
- завершённость;
- временную опору;
- автора утверждения;
- связь с другими клаузами.

### 3.2. Query Frame становится проекцией общей интерпретации

Общий языковой анализ создаёт устойчивую интерпретацию реплики. Вопросительная клауза
проецируется в существующий `Query Frame`:

```text
Clause Interpretation
→ Question Projection
→ Query Frame
→ Query Scene
→ Memory Search
→ Candidate Admission
→ Response Plan
```

### 3.3. Решения принимаются по независимым доказательствам

Каждый анализатор выпускает типизированные доказательства. Доказательства объединяются
по независимым группам:

```text
morphology
agreement
syntax
valency
semantics
discourse
temporal_context
source
manual_validation
```

Сигналы сначала агрегируются внутри группы, затем между группами.

### 3.4. Интерпретация уточняется детерминированными циклами

Один цикл содержит пять стадий:

1. морфология;
2. фразовая структура;
3. клаузы и операторы;
4. валентность и роли;
5. диалоговый контекст.

Параметры:

```text
max_interpretation_cycles = 3
stable_cycles_required = 2
```

Интерпретация останавливается раньше лимита, когда два последовательных цикла
сохраняют:

- одинаковые границы клауз;
- одинаковые главные предикаты;
- одинаковые выбранные роли;
- одинаковые референты;
- одинаковый коммуникативный режим;
- одинаковую область отрицания.

### 3.5. Память разделяется по назначению

Используются четыре слоя:

1. **Рабочая память** — активные гипотезы текущей реплики.
2. **Диалоговая эпизодическая память** — сообщения, темы, референты, вопросы, ответы и
   обязательства говорящих.
3. **Языковая память** — формы, конструкции, валентность и закономерности языка.
4. **Подтверждённая память мира** — допущенные события, сущности, отношения и
   источники.

### 3.6. Числовые показатели получают отдельные значения

```text
prior_mass  = частота наблюдения объекта или конструкции
activation  = значимость в текущей интерпретации или диалоге
support     = сила доказательств конкретной гипотезы
inertia     = устойчивость элемента в локальном фокусе
confidence  = оценка отдельного анализатора
```

Структурный допуск использует `support`, критические ограничения и независимость
доказательств. `activation` и физическая динамика ранжируют уже допущенных кандидатов.

## 6. Правила интерпретации

### 6.1. Морфология

1. Сохранять все существенные варианты `pymorphy3`.
2. Выпускать evidence для леммы, части речи и грамматических признаков.
3. Уточнять варианты через согласование, структуру фразы, валентность и дискурс.
4. Записывать выбранный вариант и сохранённые альтернативы в трассу.

### 6.2. Фразовая структура

Поддержать:

- несколько согласованных определений;
- вложенные именные группы;
- генитивные цепочки;
- приложения;
- однородные члены;
- числительные и количество;
- причастные группы;
- составные предложные операторы;
- глагольные группы.

Каждая фраза хранит:

- границы;
- голову;
- зависимости;
- грамматические признаки;
- confidence;
- evidence;
- альтернативные границы.

### 6.3. Клаузы

Определять границы по:

- предикативным центрам;
- союзам;
- пунктуации;
- инфинитивным конструкциям;
- причастным и деепричастным конструкциям;
- прямой речи;
- косвенной речи;
- эллиптическим предикатам.

### 6.4. Отрицание

`negation_scope` указывает точную цель:

```text
EVENT
PREDICATE
PARTICIPANT
ROLE_VALUE
ATTRIBUTE
QUANTITY
FREQUENCY
MODALITY
CLAUSE
```

Примеры ожидаемой структуры:

```text
Артём не ремонтирует робота.
scope = EVENT

Не Артём ремонтирует робота.
scope = PARTICIPANT(agent)

Артём ремонтирует не робота, а двигатель.
scope = ROLE_VALUE(object=робот)
asserted alternative = object=двигатель

Артём не всегда ремонтирует робота.
scope = FREQUENCY
```

### 6.5. Валентность и роли

Для каждого предиката хранить:

- обязательные роли;
- частые дополнительные роли;
- грамматические слоты;
- падежное управление;
- предлоги;
- типичные типы сущностей;
- активные и пассивные конструкции;
- возвратность;
- вид.

Роль выбирается по совокупности:

- падежа;
- предлога;
- согласования;
- порядка слов;
- залога;
- валентности;
- семантической совместимости;
- контекста.

### 6.6. Референция и эллипсис

Кандидат референта получает evidence по:

- роду;
- числу;
- одушевлённости;
- роли;
- расстоянию;
- текущей теме;
- позиции в focus stack;
- семантической совместимости;
- явному повторному именованию.

Критическая неоднозначность создаёт `pending_clarification` с двумя ведущими
кандидатами.

Эллипсис восстанавливается из:

1. активного ожидаемого ответа;
2. последнего устойчивого Query Frame;
3. последней устойчивой диалоговой сцены;
4. активной темы.

### 6.7. Исправление

`CORRECTION` содержит:

- исправляемый ход;
- исправляемую клаузу;
- исправляемую роль;
- прежнее значение;
- новое значение;
- сохранённые опоры;
- причину связи с прошлым ходом.

Исправление выполняет:

1. поиск целевого обязательства или Query Frame;
2. создание новой интерпретации;
3. перенос совместимых опор;
4. замену указанной роли;
5. маркировку старого обязательства `SUPERSEDED`;
6. пересчёт зависимого локального состояния;
7. создание `CORRECTION_ACK`.

### 6.8. Смена темы

Новая тема создаётся при явном новом главном объекте или событии без референциальной
связи с активной темой.

`topic_history` хранит:

- идентификатор темы;
- главные сущности;
- главные события;
- временную опору;
- последний активный ход;
- свёрнутое состояние.

Возврат к теме восстанавливает её свёрнутое состояние в рабочую память.

## 7. Политика памяти

### 7.1. Сырая история

Каждая пользовательская и системная реплика сохраняется с:

- говорящим;
- номером хода;
- временем;
- исходным текстом;
- версией интерпретатора;
- ссылками на акты, клаузы и гипотезы.

### 7.2. Диалоговая память

Диалоговая память хранит:

- активные темы;
- focus stack;
- вопросы и ожидаемые ответы;
- speaker commitments;
- shared confirmations;
- исправления;
- исключения;
- нерешённые ссылки;
- производные ответы.

Диалоговые объекты имеют локальные `activation` и `inertia`.

### 7.3. Языковая память

Языковая память получает:

- наблюдённые словоформы;
- морфологические варианты;
- фразовые шаблоны;
- конструкции клауз;
- валентность;
- способы выражения вопроса, условия, отрицания и цитирования.

Каждая закономерность проходит жизненный цикл:

```text
OBSERVED
→ CANDIDATE
→ PROBABLE
→ STABLE
```

Статус зависит от числа независимых подтверждённых употреблений.

### 7.4. Подтверждённая память мира

Запись проходит цепочку:

```text
Raw Source
→ Interpretation Staging
→ Structural Validation
→ Factuality Validation
→ Source Validation
→ Admission Decision
→ Confirmed Scene/Event
```

Подтверждённый объект хранит:

- сырой источник;
- документ или диалог;
- говорящего или автора;
- клаузу;
- выбранную интерпретацию;
- альтернативы;
- evidence;
- режим высказывания;
- фактуальность;
- время;
- версию парсера;
- решение о допуске.

### 7.5. Производный ответ системы

Ответ хранится со статусом `DERIVED_ANSWER` и ссылками на исходные свидетельства.
Его поддержка равна поддержке исходных источников. Число независимых источников
сохраняется без увеличения.

### 7.6. Отзыв и замена

Каждое свидетельство хранит зависимости:

```text
source
→ interpretation
→ scene/event
→ relation/concept
→ query answer
```

Операции:

- `RETRACT_EVIDENCE`;
- `SUPERSEDE_INTERPRETATION`;
- `REPROCESS_SOURCE`;
- `MERGE_ENTITY`;
- `SPLIT_ENTITY`;
- `MARK_OUTDATED`;
- `RECALCULATE_DEPENDENTS`.

## 13. Ворота качества

### Gate 1. Совместимость

- проходят текущие 114 тестов;
- простые вопросы сохраняют Query Frame;
- строгий допуск кандидатов сохраняется;
- текущая генерация проходит round-trip.

### Gate 2. Безопасность интерпретации

- вопрос маршрутизируется в Query Frame;
- условие хранится как hypothetical;
- гипотеза хранится как stated/possible;
- цитата сохраняет автора содержания;
- отрицание получает точную область действия;
- derived answer сохраняет исходное число независимых свидетельств.

### Gate 3. Связный диалог

- работают focus stack и topic history;
- работают местоимения и эллипсис;
- критическая неоднозначность создаёт уточнение;
- работает смена и возврат темы;
- работает исправление конкретного слота.

### Gate 4. Управляемая память

- глобальный объект имеет полный provenance;
- staging и commit разделены;
- quarantine доступен для проверки;
- retraction пересчитывает зависимости;
- reprocessing сохраняет историю версий.

### Gate 5. Масштабирование

- партии поддерживают preview, commit, rollback и reprocess;
- регрессия запускается до commit;
- метрики времени, памяти и базы сохраняются;
- качество старых сценариев сохраняется после каждой партии.

## 14. Первый вертикальный срез

Первый сквозной сценарий:

```text
Пользователь: Привет! Что продают на рынке?
Система: Рыбу.
Пользователь: А ещё что?
Система: В доступной памяти других товаров нет.
Пользователь: Нет, я имел в виду магазин.
Система: В магазине продают мясо и кошачий корм.
Пользователь: А удочку там продают?
Система: Да, в рыболовном магазине продают удочку.
```

Срез реализует:

1. `GREETING + QUESTION` в одной реплике.
2. Query Frame с `location=рынок` и пустым `object`.
3. Ответ из подтверждённой памяти.
4. Продолжение с исключением прошлого ответа.
5. Ответ `UNKNOWN` при отсутствии второго кандидата.
6. `CORRECTION` места и типа вопроса.
7. Новый Query Frame с `location=магазин` и пустым `object`.
8. Обновление active topic и focus stack.
9. Разрешение `там`.
10. Полярный вопрос.
11. `DERIVED_ANSWER` с provenance.
12. Полную трассу актов, клауз, evidence, состояния и ответа.

Критерии готовности среза:

- все двенадцать пунктов отражены в данных;
- каждый переход воспроизводим;
- состояние восстанавливается после перезапуска;
- текущие тесты V2 проходят;
- сценарий входит в постоянную регрессию.

## 15. Definition of Done V2.5

V2.5 завершена, когда система:

1. Разбирает составную реплику на несколько актов и клауз.
2. Извлекает несколько событий и связи между ними.
3. Хранит независимые оси режима, фактуальности, полярности, модальности и
   завершённости.
4. Определяет область отрицания.
5. Хранит конкурирующие гипотезы и evidence.
6. Выполняет максимум три детерминированных цикла интерпретации.
7. Завершает устойчивую интерпретацию после двух одинаковых циклов.
8. Возвращает `AMBIGUOUS`, `INCOMPLETE` или `CONFLICTED` для соответствующих случаев.
9. Строит Query Frame из общей интерпретации вопроса.
10. Поддерживает focus stack, pending questions и topic history.
11. Разрешает основные местоимения и эллипсис.
12. Формирует уточнение по спорному слоту.
13. Поддерживает исправление прошлого хода.
14. Различает speaker commitment, shared confirmation, world knowledge и
    derived answer.
15. Хранит языковую память отдельно от подтверждённой памяти мира.
16. Проводит знание через staging и admission.
17. Поддерживает quarantine, retraction и reprocessing.
18. Формирует все типы `ResponsePlan`.
19. Выполняет обратную проверку ролей, полярности, модальности, фактуальности и
    атрибуции.
20. Объясняет путь от исходной реплики до ответа.
21. Сохраняет совместимость API V2.
22. Проходит текущую регрессию, диалоговый эталон и все ворота качества.
